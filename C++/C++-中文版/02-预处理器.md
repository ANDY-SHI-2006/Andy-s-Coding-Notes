# 2 预处理器

预处理器在编译之前运行，执行文本替换和条件包含。

## 2.1 `#include`：头文件包含机制

`#include` 执行**文本替换**——整个头文件的内容被插入到指令所在的位置。

```cpp
#include <iostream>   // 搜索系统包含路径
#include "myheader.h" // 先搜索项目目录，再搜索系统路径
```

### 2.1.1 搜索路径：`< >` 与 `" "`

| 语法 | 搜索顺序 | 使用场景 |
| ---- | -------- | -------- |
| `<header>` | 仅系统目录 | 标准库头文件（`<iostream>`、`<vector>`） |
| `"header"` | 当前目录 → 项目路径 → 系统目录 | 自定义/项目头文件（`"utils.h"`、`"config.h"`） |

**关键区别详解：**

```cpp
#include <iostream>     // 标准 C++ 头文件——仅搜索系统路径
#include "myclass.hpp"  // 项目头文件——先搜索本地目录
#include <cmath>        // 标准 C++ 数学头文件
#include "math_helper.h" // 自定义数学工具——先搜索本地目录
```

**系统路径（典型位置）：**
- Linux: `/usr/include`, `/usr/local/include`
- Windows (MinGW): `C:\MinGW\include`
- macOS: `/Applications/Xcode.app/Contents/Developer/Platforms/...`

> **注意：** 虽然 `"header"` 搜索更多位置，但 `<header>` 在语义上才是标准库头文件的正确写法。系统/标准头文件用 `<>`，自己的头文件用 `""`。

### 2.1.2 C 与 C++ 头文件：`<cstdio>` 与 `<stdio.h>`

C++ 提供两种包含 C 标准库头文件的方式：

| 风格 | 示例 | 命名空间 | 推荐程度 |
| ---- | ---- | -------- | -------- |
| C 风格 | `#include <stdio.h>` | 全局命名空间 | 遗留/C 兼容 |
| C++ 风格 | `#include <cstdio>` | `std::` 命名空间（实现可能也放入全局） | **C++ 中优先使用** |

**完整映射表：**

| C 头文件 | C++ 等价头文件 | 内容 |
|----------|----------------|------|
| `<stdio.h>` | `<cstdio>` | 文件 I/O（`printf`、`scanf`、`FILE`） |
| `<stdlib.h>` | `<cstdlib>` | 通用工具（`malloc`、`free`、`rand`、`exit`） |
| `<string.h>` | `<cstring>` | C 字符串操作（`strcpy`、`strlen`、`strcmp`） |
| `<ctype.h>` | `<cctype>` | 字符分类（`isdigit`、`isalpha`、`toupper`） |
| `<math.h>` | `<cmath>` | 数学函数（`sin`、`cos`、`sqrt`、`pow`） |
| `<time.h>` | `<ctime>` | 时间函数（`time`、`clock`、`strftime`） |
| `<assert.h>` | `<cassert>` | 运行时断言（`assert`） |
| `<errno.h>` | `<cerrno>` | 错误编号（`errno`） |
| `<limits.h>` | `<climits>` | 整数类型限制（`INT_MAX`、`LONG_MIN`） |
| `<float.h>` | `<cfloat>` | 浮点数限制（`FLT_MAX`、`DBL_EPSILON`） |
| `<stddef.h>` | `<cstddef>` | 通用定义（`size_t`、`NULL`、`nullptr_t`） |

**关键区别：**

```cpp
#include <stdio.h>    // C 风格：名称放入全局命名空间
printf("Hello\n");     // 可直接使用

#include <cstdio>     // C++ 风格：名称放入 std:: 命名空间
std::printf("Hello\n"); // 推荐：显式命名空间
// 或者
using namespace std;
printf("Hello\n");    // 也能工作（但在头文件中使用 using namespace std 不推荐）
```

> **最佳实践：** 新 C++ 代码始终使用 C++ 风格头文件（`<cxxx>`）。它们将名称正确放入 `std::` 命名空间，避免命名空间污染并遵循 C++ 惯例。

### 2.1.3 头文件扩展名

| 扩展名 | 惯例 | 典型内容 |
|--------|------|----------|
| `.h` | C 头文件，部分 C++ 遗留 | C 标准库、操作系统 API |
| `.hpp` / `.hh` / `.h++` | C++ 专属 | 模板、类、C++ 特性 |

> **注意：** `.hpp`、`.hh` 和 `.h++` 是等价的惯例——都表示 C++ 专属头文件。`.hpp` 使用最广泛。

> **最佳实践：** C++ 头文件使用 `.hpp` 扩展名，以区别于 C 头文件。

## 2.2 宏定义 (#define)

**宏的本质：文本替换**

预处理器执行**纯文本替换**——它字面地将宏名称替换为定义的文本，不做任何类型检查、语法分析或语义理解。

```cpp
#define PI 3.14159
#define SQUARE(x) x * x

// 预处理前（你写的代码）
double area = PI * SQUARE(r);

// 预处理后（编译器看到的代码）
double area = 3.14159 * r * r;
```

**关键含义：**
- **无类型安全**：预处理器不了解也不关心 C++ 类型
- **无作用域**：宏不是变量；它们从 `#define` 开始到 `#undef` 或文件末尾一直存在
- **不占用内存**：宏不分配存储空间；它们在编译前就被替换了
- **简单但危险**：如果不小心，文本替换可能产生意外结果（见下文陷阱）

### 2.2.1 类对象宏（常量）

```cpp
#define PI 3.141593
#define MAX_SIZE 100

// 使用方式
double area = PI * radius * radius;  // 编译器看到：3.141593 * radius * radius
int arr[MAX_SIZE];                    // 编译器看到：int arr[100];
```

> **现代替代方案：** 尽可能使用 `const` 变量或 `constexpr` 代替宏：

```cpp
 constexpr double PI = 3.141593;  // 类型安全，尊重作用域 
```
 
### 2.2.2 类函数宏

```cpp
#define SQUARE(x) ((x) * (x))
#define MAX(a, b) ((a) > (b) ? (a) : (b))

// 使用方式
int y = SQUARE(5);      // 展开为：((5) * (5)) = 25
int m = MAX(x, y);      // 展开为：((x) > (y) ? (x) : (y))
```

### 2.2.3 多行宏

如果宏定义跨越多行，必须在每行末尾（最后一行除外）放置反斜杠（`\`）以表示行继续。

```cpp
// 使用反斜杠继续的多行宏
#define PRINT_RECTANGLE(width, height)     \
    for (int i = 0; i < height; i++) {     \
        for (int j = 0; j < width; j++) {  \
            std::cout << "*";              \
        }                                  \
        std::cout << std::endl;            \
    }
```

**重要规则：**
- 反斜杠必须是该行的**最后一个字符**（后面不能有空格或注释）
- 反斜杠转义换行符，因此预处理器将其视为一行连续的代码
- 反斜杠后的缩进仅为了可读性

**常见错误：**

反斜杠 `\` 必须是该行的**最后一个字符**。如果后面有空格，行继续就会中断：

```cpp
// 错误——反斜杠后有空格
#define BAD_MACRO(x) x + \   
    1
```

**失败原因：** 预处理器看到的是 `\[空格]`——`\` 只转义了空格，而没有转义换行符。宏在第一行结束，第二行变成无效的独立代码。

```cpp
// 正确——\ 后面没有任何字符
#define GOOD_MACRO(x) x + \
    1
```

### 2.2.4 宏展开规则与陷阱

**关键规则：所有内容都要加括号**

```cpp
// 错误——缺少括号
#define SQUARE(x) x * x

SQUARE(a + b);          // 展开为：a + b * a + b  （错了！）
// 应该是：(a + b) * (a + b)

// 正确
#define SQUARE(x) ((x) * (x))

// 错误——缺少外层括号
#define DOUBLE(x) (x) + (x)

DOUBLE(5) * 3;          // 展开为：(5) + (5) * 3 = 20  （错了！）
// 应该是：((5) + (5)) * 3 = 30

// 正确
#define DOUBLE(x) ((x) + (x))
```

**常见陷阱：多次求值**

```cpp
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int x = 5;
int m = MAX(x++, 10);   // 展开为：((x++) > (10) ? (x++) : (10))
// x 被自增了两次！
```

**最佳实践：** 在现代 C++ 中使用内联函数代替类函数宏：
```cpp
inline int max(int a, int b) { return a > b ? a : b; }  // 只求值一次，类型安全
```

与宏不同，`inline` 函数：
- 参数**只求值一次**（没有双自增 bug）
- 具有**类型检查**并尊重作用域规则
- 是**真正的函数**，调试器可以看到

> 关于 `inline` 的工作原理及使用时机，详见[内联函数](09-函数.md#926-inline-functions)。

### 2.2.5 宏与函数：总结

**宏相对于函数的优势：**

| 方面 | 宏 | 函数 |
|------|-----|------|
| **定义** | 单行 `#define`，无需分离 | 头文件中声明，源文件中定义 |
| **编译** | 预处理期文本替换 | 生成函数调用指令 |
| **链接** | 无链接开销（已经展开） | 需要符号解析 |
| **执行** | 无运行时开销（默认内联） | 跳转、执行、返回 |
| **类型检查** | 无（文本替换） | 完整类型检查 |

**宏的跨文件使用：**

- **不能直接跨文件使用：** 在 `.cpp` 文件中定义的宏仅在该文件内可见
- **可以通过头文件跨文件使用：** 将宏定义放在头文件（`.hpp`）中，然后在每个需要它的源文件中 `#include`

```cpp
// math_utils.hpp
#ifndef MATH_UTILS_HPP
#define MATH_UTILS_HPP
#define SQUARE(x) ((x) * (x))  // 通过头文件共享
#endif

// file1.cpp
#include "math_utils.hpp"  // 此处可使用 SQUARE

// file2.cpp  
#include "math_utils.hpp"  // 此处也可使用 SQUARE
```

> **要点：** 每个 `.cpp` 文件通过 `#include` 获得宏的独立副本；宏不像函数那样被"共享"，而是被预处理器"复制粘贴"。

### 2.2.6 宏运算符（高级）

预处理器为宏提供了两个特殊运算符：`#`（字符串化）和 `##`（标记拼接）。

#### 2.2.6.1 `#` 运算符（字符串化）

将宏参数转换为字符串字面量。

```cpp
#define STRINGIFY(x) #x

std::cout << STRINGIFY(Hello World);  // 展开为："Hello World"
std::cout << STRINGIFY(123 + 456);    // 展开为："123 + 456"
```

**应用场景：** 带变量名的调试日志
```cpp
#define DEBUG_VAR(var) std::cout << #var " = " << var << std::endl

int count = 42;
DEBUG_VAR(count);  // 输出：count = 42
```

#### 2.2.6.2 `##` 运算符（标记拼接）

将两个标记拼接为一个标记。
```cpp
#define CONCAT(a, b) a##b

int xy = 10;
std::cout << CONCAT(x, y);  // 展开为：xy（该变量）
```

**应用场景：** 程序化地生成变量名或函数名
```cpp
#define DECLARE_VARIABLE(type, name) type var_##name

DECLARE_VARIABLE(int, count);   // 展开为：int var_count
DECLARE_VARIABLE(double, pi);   // 展开为：double var_pi
```

#### 2.2.6.3 组合使用两种运算符

```cpp
#define MAKE_FUNCTION(name) void func_##name() { std::cout << #name " called\n"; }

MAKE_FUNCTION(init);    // 展开为：void func_init() { std::cout << "init called\n"; }
MAKE_FUNCTION(cleanup); // 展开为：void func_cleanup() { std::cout << "cleanup called\n"; }
```

> **注意：** 这些运算符是高级特性。现代 C++ 通常更倾向于使用模板和 constexpr 函数以获得类型安全。


### 2.2.7 `#undef` —— 移除宏定义

```cpp
#define MAX_SIZE 100
// ... 使用 MAX_SIZE 的代码 ...
#undef MAX_SIZE
// MAX_SIZE 不再被定义
#define MAX_SIZE 200  // 可以用新值重定义
```


## 2.3 条件编译

条件编译是**预处理器的 if-else 版本**。在编译前，预处理器评估条件并决定保留哪些代码、丢弃哪些代码。

### 2.3.1 `#ifdef`、`#ifndef`、`#if` defined()

这些指令的工作方式类似于 `if` 语句，但它们在**预处理期**求值，而非运行时。

**第 1 步：`#ifdef` —— "如果已定义"**

```cpp
#define DEBUG

#ifdef DEBUG
    std::cout << "Debug mode" << std::endl;
#endif
```

`#ifdef DEBUG` 检查 `DEBUG` 是否已被 `#define`。如果是，内部代码被保留；否则，预处理器在编译前删除整个代码块。

**第 2 步：`#ifndef` —— "如果未定义"**

```cpp
#ifndef VERSION
    #define VERSION "1.0"
#endif
```

`#ifndef` 与之相反——仅当宏**未**被定义时，才编译内部代码。这是包含守卫（Include Guard）的核心机制。

**第 3 步：`#if defined()` —— 更灵活的形式**

```cpp
#if defined(DEBUG)       // 等价于 #ifdef DEBUG
    // 调试代码
#endif

#if !defined(NDEBUG)     // 等价于 #ifndef NDEBUG
    // 调试代码（如果 NDEBUG 未定义）
#endif
```

`defined()` 可以组合多个条件（例如 `#if defined(WIN32) && defined(DEBUG)`），而 `#ifdef` 无法做到这一点。

**对比表：**

| 指令 | 含义 | C++ 等价写法 |
|------|------|-------------|
| `#ifdef X` | 如果 X 已定义则编译 | `if (X 存在)` |
| `#ifndef X` | 如果 X 未定义则编译 | `if (X 不存在)` |
| `#if defined(X)` | 等价于 `#ifdef X` | `if (X 存在)` |
| `#if !defined(X)` | 等价于 `#ifndef X` | `if (X 不存在)` |
| `#elif` | 否则如果 | `else if` |
| `#else` | 否则 | `else` |
| `#endif` | 条件块结束 | `}` |

### 2.3.2 平台检测

不同平台需要不同的代码。预处理器可以在编译期检测操作系统：

```cpp
#if defined(_WIN32)
    // Windows（32 位和 64 位）
    #include <windows.h>
#elif defined(__APPLE__)
    // macOS
    #include <TargetConditionals.h>
    #if TARGET_OS_MAC
        // macOS 专属代码
    #endif
#elif defined(__linux__)
    // Linux
    #include <unistd.h>
#elif defined(__unix__)
    // Unix（通用）
#endif
```

**常用平台宏：**

| 宏 | 平台 |
|----|------|
| `_WIN32` | Windows（32 位和 64 位） |
| `_WIN64` | 仅 Windows 64 位 |
| `__APPLE__` | macOS、iOS |
| `__linux__` | Linux |
| `__unix__` | Unix（Linux、macOS、BSD） |
| `__FreeBSD__` | FreeBSD |
| `__ANDROID__` | Android |

### 2.3.3 Debug 与 Release 构建

开发者在开发期间需要调试输出，但在发布程序时希望将其移除。条件编译可以自动处理这一点：

```cpp
#ifdef DEBUG
    #define LOG(msg) std::cout << "[DEBUG] " << msg << std::endl
#else
    #define LOG(msg)    // Release 中移除
#endif

LOG("Entering function foo");
```

**预处理后：**

| 构建 | 命令 | `LOG(...)` 变成 |
|------|------|----------------|
| Debug | `g++ -DDEBUG main.cpp` | `std::cout << "[DEBUG] " << ...` |
| Release | `g++ main.cpp` | （无） |

**配合断言检查：**

```cpp
#ifdef DEBUG
    #define LOG(msg) std::cout << "[DEBUG] " << msg << std::endl
    #define ASSERT(cond) if(!(cond)) { std::cerr << "Assertion failed"; exit(1); }
#else
    #define LOG(msg)
    #define ASSERT(cond)
#endif

LOG("Pointer allocated");
ASSERT(ptr != nullptr);
```

## 2.4 包含守卫

包含守卫防止同一个头文件被多次包含。没有它们的话，多次包含头文件会导致**重定义错误**。

### 2.4.1 问题：多次包含

考虑以下场景：

```cpp
// utils.hpp
class Helper { ... };

// a.hpp
#include "utils.hpp"    // 此处定义 Helper 类

// b.hpp  
#include "utils.hpp"    // 再次定义 Helper 类

// main.cpp
#include "a.hpp"        // 通过 utils.hpp 定义 Helper
#include "b.hpp"        // 通过 utils.hpp 再次定义 Helper
// 错误：class Helper 重定义
```

当 `main.cpp` 同时包含 `a.hpp` 和 `b.hpp` 时，`utils.hpp` 被处理了两次。由于 C++ 不允许同一个类被多次定义，这会导致编译错误。

### 2.4.2 传统包含守卫（`#ifndef` / `#define` / `#endif`）

经典方法使用预处理器指令确保头文件内容只被处理一次：

```cpp
// myheader.hpp
#ifndef MYHEADER_HPP
#define MYHEADER_HPP

// 头文件内容在此
class MyClass { ... };

#endif // MYHEADER_HPP
```

**工作原理：**

| 步骤 | 第一次包含 | 第二次包含 |
| ---- | ---------- | ---------- |
| 1 | `#ifndef MYHEADER_HPP` 检查宏是否已定义 | 同样检查 |
| 2 | 宏**未**定义 → 条件为真 | 宏**已**定义 → 条件为假 |
| 3 | `#define MYHEADER_HPP` 创建宏 | 整个代码块被跳过 |
| 4 | 头文件内容被处理 | 什么都不发生 |

**命名惯例：**
- 使用大写文件名 + `_HPP` 后缀
- 示例：`math_utils.hpp` → `MATH_UTILS_HPP`
- 确保唯一性以避免名称冲突

### 2.4.3 `#pragma once`

一种更现代、更简洁的替代方案：

```cpp
// myheader.hpp
#pragma once

// 头文件内容在此
class MyClass { ... };
```

**工作原理：**
编译器内部记录哪些文件已被包含。再次遇到同一个文件时，完全跳过处理。

### 2.4.4 对比

| 方面 | `#ifndef` 守卫 | `#pragma once` |
| ---- | -------------- | -------------- |
| **可移植性** | 100% ISO C++ 标准 | 所有主流编译器支持（GCC、Clang、MSVC），但非标准 |
| **性能** | 较慢（必须打开文件检查宏） | 较快（编译器记录已包含文件） |
| **样板代码** | 3 行代码 | 1 行代码 |
| **潜在问题** | 如果宏名不唯一，可能冲突 | 无 |
| **使用场景** | 最大兼容性 | 现代项目 |

### 2.4.5 推荐

- **新项目：** 使用 `#pragma once`（更简洁、更快）
- **最大可移植性：** 使用 `#ifndef` 守卫
- **一致性很重要：** 选择一种方法并在整个项目中统一使用


## 2.5 预定义宏

预处理器提供了包含编译期信息的宏。

### 2.5.1 标准宏

| 宏 | 描述 | 示例值 |
|----|------|--------|
| `__FILE__` | 当前源文件名 | `"main.cpp"` |
| `__LINE__` | 当前行号 | `42` |
| `__func__` | 当前函数名 | `"main"` |
| `__DATE__` | 编译日期 | `"Apr 4 2026"` |
| `__TIME__` | 编译时间 | `"15:30:00"` |
| `__cplusplus` | C++ 标准版本 | `202002L` (C++20) |

```cpp
void log_error(const char* msg) {
    std::cerr << "Error in " << __FILE__ << ":" << __LINE__
              << " (" << __func__ << "): " << msg << std::endl;
}
```

**使用 `__cplusplus` 编写版本专属代码：**

```cpp
#if __cplusplus >= 202002L
    // C++20 或更高版本
    using std::format;
#elif __cplusplus >= 201703L
    // C++17
    // 使用 fmt 库或 printf
#else
    #error "C++17 or later required"
#endif
```

### 2.5.2 编译器/平台识别

| 宏 | 含义 |
|----|------|
| `__GNUC__` | GCC 编译器版本（主版本号） |
| `__clang__` | Clang 编译器 |
| `_MSC_VER` | Microsoft Visual C++ 版本 |
| `__STDC__` | 标准 C 兼容性 |

**检测编译器以编写平台专属代码：**

```cpp
#if defined(__GNUC__) && !defined(__clang__)
    // GCC 专属代码
    #pragma GCC optimize("O3")
#elif defined(__clang__)
    // Clang 专属代码
    #pragma clang optimize on
#elif defined(_MSC_VER)
    // Microsoft Visual C++ 专属代码
    #pragma warning(disable: 4996)
#endif
```

## 2.6 其他指令

除了 `#include`、`#define` 和条件编译之外，C++ 还提供了几个专门的预处理器指令用于高级场景。

| 指令 | 作用 | 常见用途 |
|------|------|----------|
| `#line` | 覆盖行号和文件名 | 代码生成器、调试生成代码 |
| `#pragma` | 编译器专属指令 | 包含守卫、优化、警告 |
| `#error` | 强制编译错误 | 版本检查、强制要求 |
| `#warning` | 发出编译期警告 | 可移植性提示、弃用特性 |

### 2.6.1 `#line` —— 调试生成代码

**作用：** 在错误消息中改变显示的行号和文件名。

**为什么使用它？**
当你编写代码生成器（例如语法分析器生成器、IDL 编译器）时，生成的 C++ 代码可能有错误。通过使用 `#line`，错误消息指向**原始源文件**而非生成后的代码。

**语法：**
```cpp
#line number "filename"
```

**示例：**
```cpp
// 原始文件：calculator.y（yacc/bison 语法文件）
// 生成文件：calculator.cpp

#line 42 "calculator.y"
int parse_expression() {
    int x = 5
    // 此处的错误将显示为："calculator.y:42" 而非 "calculator.cpp:123"
}
```

**不使用 `#line`：**
```
calculator.cpp:156: error: expected ';' before 'return'
```

**使用 `#line`：**
```
calculator.y:42: error: expected ';' before 'return'
```

> **使用时机：** 仅在编写生成 C++ 代码的代码生成器或预处理工具时使用。

### 2.6.2 `#pragma` —— 编译器指令

**作用：** 向编译器发送实现专属的命令。

> **注意：** `#pragma` 指令是非标准化的（`STDC` pragma 除外）。不同编译器支持不同的 pragma，但许多常见 pragma 在 GCC、Clang 和 MSVC 中都能工作。

#### 常用 `#pragma` 指令

| Pragma | 效果 | 可移植性 |
|--------|------|----------|
| `#pragma once` | 包含守卫（防止多次包含） | 通用（所有现代编译器） |
| `#pragma pack(n)` | 控制结构体成员对齐 | GCC、Clang、MSVC |
| `#pragma warning(...)` | 启用/禁用警告 | 主要为 MSVC |
| `#pragma GCC ...` | GCC 专属选项 | 仅 GCC |
| `#pragma clang ...` | Clang 专属选项 | 仅 Clang |

#### 2.6.2.1 `#pragma once` —— 包含守卫

最常见的 `#pragma`，用作 `#ifndef` 包含守卫的替代方案：

```cpp
// myheader.hpp
#pragma once

// 头文件内容在此
class MyClass { ... };
```

**对比：**

| 方面 | `#ifndef` 守卫 | `#pragma once` |
|------|--------------|----------------|
| 标准性 | ISO C++ 标准 | 非标准（但通用） |
| 速度 | 较慢（必须打开文件） | 较快（编译器追踪文件） |
| 安全性 | 可能名称冲突 | 无（基于文件） |
| 冗长度 | 3+ 行 | 1 行 |

> **推荐：** 新项目使用 `#pragma once`。它更简洁、更快，且所有主流编译器都支持（GCC、Clang、MSVC）。

#### 2.6.2.2 `#pragma pack` —— 结构体紧凑排列

控制结构体成员的内存对齐。通常，编译器会添加填充字节以提高性能：

```cpp
// 普通结构体（带填充）
struct Normal {
    char c;     // 1 字节 + 3 字节填充
    int i;      // 4 字节
};              // 总计：8 字节

// 紧凑结构体（无填充）
#pragma pack(push, 1)    // 保存当前设置，设置 pack=1
struct Packed {
    char c;     // 1 字节（无填充）
    int i;      // 4 字节
};              // 总计：5 字节
#pragma pack(pop)        // 恢复之前的设置

static_assert(sizeof(Normal) == 8);
static_assert(sizeof(Packed) == 5);
```

> **应用场景：** 二进制文件格式、网络协议、硬件寄存器——这些场景要求精确的布局。可能因未对齐内存访问而影响性能。

#### 2.6.2.3 `#pragma warning` —— MSVC 警告控制

Microsoft Visual C++ 专属（GCC/Clang 使用不同的机制）：

```cpp
// 禁用特定警告
#pragma warning(disable: 4996)  // 弃用函数警告

// 启用警告
#pragma warning(enable: 4244)   // 可能的数据丢失

// 压入/弹出警告状态
#pragma warning(push)
#pragma warning(disable: 4100)  // 未使用的参数
void func(int unused) { }
#pragma warning(pop)            // 恢复之前的状态
```

### 2.6.3 `#error` 与 `#warning` —— 编译期消息

**作用：** 在编译期间发出用户自定义消息。

| 指令 | 效果 | 编译是否继续？ |
|------|------|--------------|
| `#error "message"` | 致命错误，停止编译 | 否 |
| `#warning "message"` | 警告，继续编译 | 是（非标准） |

#### 2.6.3.1 `#error` —— 强制要求

使用 `#error` 在要求未满足时停止编译：

```cpp
// 要求 C++ 编译器
#ifndef __cplusplus
    #error "This file must be compiled as C++"
#endif

// 要求最低 C++ 版本
#if __cplusplus < 201703L
    #error "C++17 or later required. Use -std=c++17 flag."
#endif

// 要求特定平台
#if !defined(_WIN32) && !defined(__linux__)
    #error "Unsupported platform"
#endif

// 要求特定编译器特性
#ifndef __cpp_concepts
    #error "Compiler does not support C++20 concepts"
#endif
```

#### 2.6.3.2 `#warning` —— 可移植性提示

> **注意：** `#warning` 不是 ISO 标准 C++，但 GCC、Clang 和 MSVC 都支持。

```cpp
// 提示平台专属行为
#if sizeof(long) != 8
    #warning "long is not 64-bit on this platform. Code may not be portable."
#endif

// 标记弃用的代码路径
#ifdef LEGACY_MODE
    #warning "Using legacy mode. This will be removed in v2.0."
#endif

// 性能相关提示
#if defined(DEBUG) && defined(ENABLE_OPTIMIZATION)
    #warning "Optimization enabled in debug build. Debugging may be affected."
#endif
```

**跨平台 `#warning`：**
```cpp
// 为最大可移植性，改用 pragma
#if defined(_MSC_VER)
    #pragma message("Using MSVC")
#elif defined(__GNUC__)
    #warning "Using GCC"
#endif
```

### 2.6.4 总结

| 指令 | 使用时机 | 示例 |
|------|----------|------|
| `#line` | 编写代码生成器 | 将错误映射到原始源文件 |
| `#pragma once` | 头文件 | 现代包含守卫 |
| `#pragma pack` | 二进制协议 | 控制结构体布局 |
| `#error` | 版本检查 | 要求 C++17+ |
| `#warning` | 可移植性提示 | 提示 32 位限制 |

**要点：**
1. **`#pragma once`** —— 所有新头文件都使用它
2. **`#error`** —— 用于强制编译期要求
3. **`#line`** —— 仅在代码生成工具中需要
4. **其他 `#pragma`** —— 谨慎使用，尽可能优先使用标准 C++
