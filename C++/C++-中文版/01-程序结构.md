# 1 程序结构

## 1.1 翻译单元模型

### 1.1.1 从源代码到可执行文件

C++ 程序的构建遵循三阶段模型：

| 阶段 | 工具 | 输入 | 输出 | 所做工作 |
|------|------|------|------|----------|
| **预处理** | 预处理器 (cpp) | `.cpp` 源文件 + 头文件 | 翻译单元 | 处理 `#include`、`#define`、条件编译 |
| **编译** | 编译器 (g++, clang++) | 翻译单元 | 目标文件 (`.o`/`.obj`) | 生成机器码，检查语法和类型 |
| **链接** | 链接器 (ld) | 目标文件 + 库文件 | 可执行文件 | 解析符号，合并为单一程序 |

```cpp
// main.cpp
#include <iostream>      // 预处理器：文本插入 iostream 的内容
#define MAX 100          // 预处理器：将所有 MAX 替换为 100

int main() {
    std::cout << MAX;    // 编译器看到的是：std::cout << 100;
    return 0;
}
```

### 1.1.2 翻译单元与唯一定义规则

每个 `.cpp` 源文件（经过预处理后）成为一个**翻译单元**。

**核心概念：**
- 每个翻译单元独立编译
- 链接器将所有翻译单元合并为一个程序
- 定义（函数、全局变量）只能出现在**一个**翻译单元中

```cpp
// file1.cpp
int global = 10;          // 定义（分配内存）

// file2.cpp
extern int global;        // 声明（不分配内存，引用 file1 中的定义）
int global = 20;          // 错误！重定义违反了 ODR
```

> **唯一定义规则（ODR）：** 每个变量/函数/类在所有翻译单元中只能定义一次。允许多次声明。

### 1.1.3 预处理器概览

在编译器处理代码之前，**预处理器**会先执行一遍，处理以 `#` 开头的指令，这些指令指示预处理器在编译前修改源代码。

**关键指令：**

| 指令 | 作用 |
|------|------|
| `#include` | 插入另一个文件的内容 |
| `#define` | 定义宏（文本替换） |
| `#undef` | 移除宏定义 |
| `#if` / `#ifdef` / `#ifndef` | 条件编译 |
| `#pragma` | 编译器专属指令 |

**处理流程：**

```
源文件 (.cpp)
       ↓
预处理器处理 #include、#define、#if 等
       ↓
修改后的源文件（翻译单元）
       ↓
编译器生成目标代码
       ↓
链接器将目标文件合并为可执行文件
```

> **要点：** 预处理器不了解 C++ 语法、类型或作用域。它只进行纯文本替换。

**示例：**
```cpp
#define PI 3.14159
#define SQUARE(x) ((x) * (x))

// 预处理前（你写的代码）：
double area = PI * SQUARE(r);

// 预处理后，编译器看到的是：
double area = 3.14159 * ((r) * (r));
```

预处理器将在 [第 2 章](02-预处理器.md) 中详细介绍。

#### 1.1.4.3 C 与 C++ 头文件对比

C++ 提供两种包含 C 标准库头文件的方式。理解它们之间的区别对编写地道 C++ 代码非常重要。

| 风格 | 示例 | 命名空间 | 使用场景 |
|------|------|----------|----------|
| C++ 风格 | `<cstdio>` | `std::` 命名空间（标准） | 新 C++ 代码**优先使用** |
| C 风格 | `<stdio.h>` | 全局命名空间 | 遗留兼容、C 互操作 |

**完整的 C 到 C++ 头文件映射：**

| C 头文件 | C++ 等价头文件 | 内容 |
|----------|----------------|------|
| `<stdio.h>` | `<cstdio>` | 文件 I/O（`printf`、`scanf`、`FILE`） |
| `<stdlib.h>` | `<cstdlib>` | 通用工具（`malloc`、`free`、`rand`、`exit`） |
| `<string.h>` | `<cstring>` | C 字符串操作（`strcpy`、`strlen`、`strcmp`） |
| `<ctype.h>` | `<cctype>` | 字符分类（`isdigit`、`isalpha`、`toupper`） |
| `<math.h>` | `<cmath>` | 数学函数（`sin`、`cos`、`sqrt`、`pow`） |
| `<time.h>` | `<ctime>` | 时间函数（`time`、`clock`、`strftime`） |
| `<assert.h>` | `<cassert>` | 运行时断言（`assert`） |
| `<errno.h>` | `<cerrno>` | 错误编号（`errno`） |
| `<limits.h>` | `<climits>` | 整数类型限制（`INT_MAX`、`LONG_MIN`） |
| `<float.h>` | `<cfloat>` | 浮点数限制（`FLT_MAX`、`DBL_EPSILON`） |
| `<stddef.h>` | `<cstddef>` | 通用定义（`size_t`、`NULL`、`nullptr_t`） |
| `<stdint.h>` | `<cstdint>` | 定宽整数类型（`int32_t`、`uint64_t`） |

**关键区别：**

```cpp
#include <stdio.h>    // C 风格：名称位于全局命名空间
printf("Hello\n");      // 可直接使用

#include <cstdio>     // C++ 风格：名称位于 std:: 命名空间
std::printf("Hello\n"); // 推荐：显式命名空间

// 或者将特定名称引入作用域
using std::printf;
printf("Hello\n");      // 也可接受
```

> **最佳实践：** 新 C++ 代码始终使用 C++ 风格头文件（`<cxxx>`）。它们将名称正确放入 `std::` 命名空间，遵循 C++ 惯例并避免命名空间污染。仅在需要全局命名空间名称的旧版 C 代码互操作时使用 C 风格头文件（`<xxx.h>`）。


## 1.2 程序入口点：main()

每个 C++ 程序有且仅有一个入口点：`main()` 函数。

### 1.2.1 返回值与退出状态

```cpp
int main() {
    // ... 程序逻辑 ...
    return 0;  // 0 = 成功，非零 = 错误
}
```

| 声明方式 | 标准性 | 用法 |
| -------- | ------ | ---- |
| `int main()` | ✓ 标准 | **始终使用此形式** |
| `void main()` | ✗ 非标准 | 避免使用（不可移植） |

**退出状态的含义：**

| 返回值 | 约定含义 |
|--------|----------|
| `0` | 成功 |
| `1` | 一般错误 |
| `2` | 命令行使用错误 |
| `126` | 命令不可执行 |
| `127` | 命令未找到 |

> 在 C++ 中，如果 `main()` 末尾省略 `return 0;`，它会隐式返回 0：

```cpp
int main() { }  // 隐式返回 0
```

### 1.2.2 命令行参数（argc/argv）

```cpp
int main(int argc, char* argv[]) {
    // argc：参数个数（包含程序名）
    // argv：参数向量（C 字符串数组）
    
    std::cout << "程序：" << argv[0] << std::endl;
    
    for (int i = 1; i < argc; i++) {
        std::cout << "参数 " << i << "：" << argv[i] << std::endl;
    }
    
    return 0;
}
```

**执行示例：**
```bash
$ ./program -f input.txt -v
程序：./program
参数 1：-f
参数 2：input.txt
参数 3：-v
```

**`argc` 和 `argv` 是如何传递的：**

你不需要在运行程序时手动传入 `argc`。操作系统会自动处理：

1. **Shell 解析你的输入**：将 `./program -f input.txt -v` 按空格拆分
2. **操作系统启动程序**：将参数个数和数组传给程序
3. **C++ 运行时接收**：设置 `argc=4`，`argv=["./program", "-f", "input.txt", "-v"]`
4. **你的 `main()` 被调用**：这些值已经填充完毕

> **要点：** `argc` 由操作系统/Shell 计算，而不是你。C++ 运行时在程序启动时将它传递给 `main()`。

**参数形式（在 C++ 中全部等价）：**

| 声明方式 | 含义 |
|----------|------|
| `int main()` | 无命令行参数 |
| `int main(void)` | 无参数（与 C 兼容的显式形式） |
| `int main(int argc, char* argv[])` | 参数以数组形式接收 |
| `int main(int argc, char** argv)` | 参数以指向指针的指针形式接收 |

**重要：** `argc` 和 `argv` 必须同时出现。不能只写一个：

| 声明方式 | 是否有效？ |
|----------|-----------|
| `int main(int argc, char* argv[])` | ✓ 有效（两者都有） |
| `int main()` | ✓ 有效（两者都没有） |
| `int main(int argc)` | ✗ 无效（只有 argc） |
| `int main(char* argv[])` | ✗ 无效（只有 argv） |

> **规则：** 命令行参数接口由 C++ 标准固定——要么两个参数都有，要么都没有。

## 1.3 命名空间

命名空间通过创建作用域边界来防止名称冲突。

### 1.3.1 命名空间基础

```cpp
namespace math {
    const double PI = 3.14159;
    double square(double x) { return x * x; }
}

namespace physics {
    const double PI = 3.1415926535;  // 不同的 PI，没有冲突
    double energy(double m) { return m * 299792458 * 299792458; }
}

// 使用作用域解析运算符访问
double a = math::PI;
double b = physics::PI;
```

**全局命名空间限定符 (`::`)**

在名称前加 `::` 表示引用**全局命名空间**，绕过任何同名的局部或嵌套名称：

```cpp
int value = 10;

namespace outer {
    int value = 20;

    void demo() {
        int value = 30;
        cout << value;        // 30（局部）
        cout << outer::value; // 20（命名空间）
        cout << ::value;      // 10（全局）
    }
}
```

### 1.3.2 嵌套命名空间

命名空间可以嵌套，以层次化方式组织代码。

**传统 C++98/11 风格：**
```cpp
namespace graphics {
    namespace rendering {
        void draw() { }
    }
}
```

**C++17 内联嵌套语法：**
```cpp
namespace graphics::rendering {
    void draw() { }
}

// 使用方式
graphics::rendering::draw();
```

### 1.3.3 命名空间别名

过长或嵌套过深的命名空间名称可以通过别名缩短：

```cpp
namespace graphics::rendering::opengl {
    void init() { }
}

// 创建别名
namespace gl = graphics::rendering::opengl;

gl::init();  // 等价于 graphics::rendering::opengl::init()
```

### 1.3.4 匿名命名空间

**匿名命名空间**（无名命名空间）将可见性限制在当前翻译单元（源文件）内。它是现代 C++ 中替代全局 `static` 的方案。

```cpp
namespace {
    int internal_counter = 0;  // 仅在本文件内可见

    void helper() {           // 仅在本文件内可见
        ++internal_counter;
    }
}
```

> **对比：** `static int x;` 和 `namespace { int x; }` 都能实现内部链接效果，但在现代 C++ 中优先使用匿名命名空间。

### 1.3.5 using 声明与 using 指令

**1. 始终使用完全限定名（最安全）：**
```cpp
std::cout << "Hello" << std::endl;
std::cin >> x;
```
你总是可以使用完整名称。这是最明确、不会产生歧义的方法。

**2. using 声明：** 将特定名称引入当前作用域
```cpp
using std::cout;      // 只有 cout 可以不用 std:: 前缀访问
using std::endl;

cout << "Hello" << endl;  // 可以
// cin >> x;              // 错误：cin 未声明
```

**3. using 指令：** 将整个命名空间的名称全部引入
```cpp
using namespace std;  // 所有 std 名称均可直接访问

cout << "Hello" << endl;  // 可以
cin >> x;                  // 可以
```

### 1.3.6 最佳实践

```cpp
// 糟糕：在头文件中使用
#pragma once
using namespace std;  // ✗ 污染所有包含此头文件的文件！

// 良好：在头文件中完全限定
#pragma once
#include <string>

class MyClass {
    std::string name;  // ✓ 显式限定
};
```

**最佳实践总结：**
- **切勿**在头文件（`.h`/`.hpp`）中使用 `using namespace`
- **优先**使用 `using 声明`（引入特定名称）而非 `using 指令`（引入整个命名空间）
- **限制** `using namespace` 的使用范围，尽量缩小到最小作用域（例如函数内部或 `.cpp` 文件）
- **优先**使用 `命名空间别名` 代替深层嵌套的 `using namespace`，以提高可读性
- 在 `.cpp` 文件中，小型程序使用 `using namespace std;` 是可以接受的

## 1.4 源文件组织

随着程序规模增大，将代码拆分到多个文件中变得至关重要。本节介绍如何在多个源文件中组织 C++ 代码。

### 1.4.1 双文件模型（.cpp + .hpp）

在 C++ 中，每个组件通常由两个文件组成：

| 文件类型 | 扩展名 | 包含内容 | 作用 |
| -------- | ------ | -------- | ---- |
| **头文件** | `.hpp` | 声明（函数原型、类声明） | 接口——告诉其他文件有哪些功能可用 |
| **源文件** | `.cpp` | 定义（函数实现、全局变量） | 实现——实际的代码 |

**为什么要分离？**
- **编译效率**：头文件被多次包含；源文件只编译一次
- **接口隐藏**：用户看到的是 API（头文件），而非实现细节
- **代码复用**：多个源文件可以包含同一个头文件

**示例结构：**
```
project/
├── math_utils.hpp    // 声明
├── math_utils.cpp    // 实现
└── main.cpp          // 使用 math_utils
```

### 1.4.2 内容归置原则

**头文件 (.hpp) —— 接口：**
```cpp
// math_utils.hpp
#pragma once

// 函数声明（原型）
double square(double x);
double cube(double x);

// 类声明
class Calculator {
public:
    double add(double a, double b);
    double multiply(double a, double b);
};
```

**源文件 (.cpp) —— 实现：**
```cpp
// math_utils.cpp
#include "math_utils.hpp"

double square(double x) {
    return x * x;
}

double cube(double x) {
    return x * x * x;
}

double Calculator::add(double a, double b) {
    return a + b;
}

double Calculator::multiply(double a, double b) {
    return a * b;
}
```

**切勿放入头文件的内容：**
- 函数定义（除非内联）
- 全局变量定义（改用 `extern` 声明）
- `using namespace` 指令

### 1.4.3 实际示例

完整的多文件项目：

**math_utils.hpp：**
```cpp
#pragma once

double calculateArea(double radius);
```

**math_utils.cpp：**
```cpp
#include "math_utils.hpp"
#define PI 3.14159

double calculateArea(double radius) {
    return PI * radius * radius;
}
```

**main.cpp：**
```cpp
#include <iostream>
#include "math_utils.hpp"

int main() {
    double r = 5.0;
    std::cout << "面积：" << calculateArea(r) << std::endl;
    return 0;
}
```

**编译：**
```bash
# 分别编译每个 .cpp 文件
g++ -c math_utils.cpp -o math_utils.o
g++ -c main.cpp -o main.o

# 将目标文件链接在一起
g++ math_utils.o main.o -o program

# 或者一次性编译全部
g++ math_utils.cpp main.cpp -o program
```

### 1.4.4 构建流程回顾

从源代码到可执行文件的完整旅程：

```
math.hpp ------+
               |
math.cpp ------+--> 编译 --> math.o ---+
               |                          |
main.cpp ------+                          +--> 链接 --> program
                                          |
                                       main.o
```

**要点：**
- 每个 `.cpp` 文件独立编译为 `.o`（目标）文件
- 链接器将所有目标文件合并为最终可执行文件
- 头文件不会被直接编译——它们被包含进 `.cpp` 文件中
- 修改头文件后，需要重新编译所有包含它的文件

> **注意：** 理解这种结构有助于解释为什么需要 `#include`（第 2 章）以及为什么存在唯一定义规则（第 1.1.2 节）。


### 1.4.5 构建系统与 CMake（可选）

随着项目规模超过几个文件，手动管理编译变得复杂。本节介绍 CMake，C++ 的行业标准构建系统。

#### 1.4.5.1 构建系统的必要性

**手动编译的问题：**

对于一个简单的双文件项目：
```bash
g++ -c math_utils.cpp -o math_utils.o
g++ -c main.cpp -o main.o
g++ math_utils.o main.o -o program
```

但对于一个拥有 20+ 文件、多个目录和外部库的真实项目：
```bash
# 你需要记住：
# - 哪些文件需要编译
# - 头文件的包含路径
# - 库文件的路径和链接标志
# - 编译器选项（优化、警告）
# - Debug 与 Release 的不同设置
```

**构建系统可以自动化这一切。** 你只需描述一次项目，构建系统就会处理所有细节。

**主流构建系统：**

| 构建系统 | 描述 | 平台 |
| -------- | ---- | ---- |
| **Make** | 传统 Unix 工具 | Linux/macOS |
| **MSBuild** | 微软的构建系统 | Windows |
| **Ninja** | 快速、现代的替代方案 | 跨平台 |
| **CMake** | 元构建系统（生成 Make/Ninja/MSBuild 文件） | 跨平台 |

> **为什么选择 CMake？** 它能为你的平台生成原生构建文件（Linux 上生成 Makefile，Windows 上生成 Visual Studio 项目，macOS 上生成 Xcode 项目）。一次编写，到处构建。

#### 1.4.5.2 CMake 基础

**CMake 的工作方式（三阶段）：**

**阶段 1：你的项目目录**
```
my_project/
├── CMakeLists.txt      （你编写的文件）
├── src/
│   └── main.cpp
└── include/
    └── math_utils.hpp
```
运行：`cmake ..`
↓

**阶段 2：生成构建文件**
CMake 读取 CMakeLists.txt 并创建平台专属的构建文件：
- Linux/macOS：`Makefile`
- Windows：`MyProject.sln`（Visual Studio）或 `Makefile`（MinGW）
- 可选：`build.ninja`（如使用 Ninja 生成器）

运行：`cmake --build .`
↓

**阶段 3：编译与链接**
CMake 调用底层构建工具：
- Linux/macOS：运行 `make`
- Windows：运行 `MSBuild` 或直接调用编译器
- 输出：`program` 可执行文件

**简而言之：**
```
CMakeLists.txt --(cmake ..)--> 构建文件 --(cmake --build .)--> 可执行文件
```

**最简 CMakeLists.txt：**

```cmake
# 指定最低 CMake 版本
cmake_minimum_required(VERSION 3.10)

# 项目名称和版本
project(MyProgram VERSION 1.0)

# 设置 C++ 标准
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 从源文件创建可执行文件
add_executable(program main.cpp math_utils.cpp)
```

**关键命令说明：**

| 命令 | 作用 |
|------|------|
| `cmake_minimum_required` | 确保 CMake 版本兼容性 |
| `project()` | 定义项目名称，设置变量 |
| `set(CMAKE_CXX_STANDARD 17)` | 请求使用 C++17（或 11、14、20） |
| `set(CMAKE_CXX_STANDARD_REQUIRED ON)` | 强制使用（而非可选） |
| `add_executable()` | 从源文件创建可执行目标 |

#### 1.4.5.3 多文件项目示例

本节展示一个完整的、头文件和源文件分目录存放的多文件项目。

**步骤 1：理解问题**

当代码按子目录组织时，编译器需要知道去哪里找头文件：

```cpp
// src/main.cpp
#include "math_utils.hpp"   // 这个文件在哪里？
```

如果没有额外配置，编译器找不到 `math_utils.hpp`，因为它位于 `include/` 目录，而非当前目录。

**步骤 2：项目结构**

```
my_project/
├── CMakeLists.txt          # 构建配置
├── include/                # 头文件 (.hpp)
│   └── math_utils.hpp
└── src/                    # 源文件 (.cpp)
    ├── main.cpp
    └── math_utils.cpp
```

**步骤 3：编写 CMakeLists.txt**

```cmake
cmake_minimum_required(VERSION 3.10)
project(Calculator)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 告诉编译器去哪里找头文件
include_directories(include)

# 列出所有需要编译的源文件
add_executable(calculator
    src/main.cpp
    src/math_utils.cpp
)
```

**与单文件项目的关键区别：**

| 方面 | 单文件 | 多文件、多目录 |
|------|--------|----------------|
| 头文件位置 | 同一目录 | `include/` 子目录 |
| 源文件 | 一个文件 | `src/` 中多个文件 |
| CMake 命令 | `add_executable(program main.cpp)` | `add_executable(program src/main.cpp src/other.cpp)` |
| 额外需求 | 无 | `include_directories(include)` |

**步骤 4：构建与运行**

```bash
# 创建构建目录（保持源码目录整洁）
mkdir build
cd build

# 生成构建文件
cmake ..

# 编译项目
cmake --build .

# 运行结果
./calculator          # Linux/macOS
calculator.exe        # Windows
```

**为什么需要 `include_directories(include)`？**

这告诉 CMake："编译时，将 `include/` 目录加入编译器的搜索路径。" 这等价于在 g++ 命令中添加 `-Iinclude`：

```bash
# 手动编译的等价命令：
g++ -Iinclude src/main.cpp src/math_utils.cpp -o calculator
```

**完整文件示例：**

```cpp
// include/math_utils.hpp
#ifndef MATH_UTILS_HPP
#define MATH_UTILS_HPP
int add(int a, int b);
#endif
```

```cpp
// src/math_utils.cpp
#include "math_utils.hpp"
int add(int a, int b) {
    return a + b;
}
```

```cpp
// src/main.cpp
#include <iostream>
#include "math_utils.hpp"

int main() {
    std::cout << "2 + 3 = " << add(2, 3) << std::endl;
    return 0;
}
```

#### 1.4.5.4 常用 CMake 选项

**编译器标志：**

```cmake
# 添加编译器警告（强烈推荐）
if(MSVC)
    add_compile_options(/W4 /WX)    # MSVC：所有警告，警告视为错误
else()
    add_compile_options(-Wall -Wextra -pedantic)  # GCC/Clang
endif()
```

**Debug 与 Release：**

```bash
# Debug 构建（带调试符号）
cmake -DCMAKE_BUILD_TYPE=Debug ..

# Release 构建（优化）
cmake -DCMAKE_BUILD_TYPE=Release ..
```

**平台差异：**

| 特性 | Linux/macOS | Windows (Visual Studio) |
|------|-------------|------------------------|
| 构建命令 | `make` 或 `cmake --build .` | `cmake --build .` 或在 Visual Studio 中打开 `.sln` |
| 可执行文件名 | `program` | `program.exe` |
| 默认生成器 | Makefile | Visual Studio 项目 |
| 指定生成器 | `cmake -G Ninja ..` | `cmake -G "MinGW Makefiles" ..` |

#### 1.4.5.5 何时学习 CMake

**初学者：**
- 小型项目（< 5 个文件）：使用手动 `g++` 命令
- 先专注于学习 C++ 语言本身

**中高级用户：**
- 拥有多个目录的项目
- 使用第三方库
- 跨平台开发
- 参与开源项目（大多数使用 CMake）

> **注意：** CMake 学习曲线较陡，但它是行业标准。建议在熟练掌握 C++ 基础后再学习它。

#### 1.4.5.6 总结

| 概念 | 要点 |
|------|------|
| **为什么用 CMake** | 自动化复杂项目的编译过程 |
| **工作原理** | 你写 `CMakeLists.txt`，CMake 生成构建文件 |
| **基本流程** | `cmake ..` 然后 `cmake --build .` |
| **最佳实践** | 始终使用外部构建（单独的 `build/` 目录） |
| **优先级** | 掌握 C++ 基础后再学习 |
